# Step 5.5 TDD plan — hubMembers re-export + filterByKeywordOverlap (Revision 2, full rewrite)

## Overview

Revision 1 was drafted against an assumed substrate (`hub-id:<hubId>` labels, per-card `kw:<term>` labels, a `resolveAddressToBead` helper) that does not exist in the codebase. Revision 2 reflects the verified substrate: `hubMembers` collapses to a one-line re-export of the already-shipped `listHubConstituents`, and `filterByKeywordOverlap` changes its signature so the caller passes keyword terms in directly — avoiding a reverse-lookup path that the current keyword index does not support. Total estimated impact: ~40 LOC impl, ~250 LOC tests. One file touched for impl (`keyword-index.ts`), one touched for the hub alias (`hubs.ts`), no new files.

## Why the rewrite (review C1 + C2)

Two plan-invalidating findings from the REVIEW forced a full rewrite rather than a patch:

1. **`hubMembers` is already shipped.** `listHubConstituents` at `apps/silmari-mcp/src/lib/hubs.ts:319-321` does exactly what the original plan specified: `listInbound('idea', hubId, 'derives-from', 1000)`. Hub membership is encoded as the edge label `ref:derives-from:<hubId>` on the card (not `hub-id:<hubId>` — zero grep hits for that prefix anywhere in the monorepo). The original plan's `hubMembers` implementation would have returned `[]` for every real hub. Fix: `export const hubMembers = listHubConstituents;` plus a test that pins the alias behavior.

2. **`filterByKeywordOverlap` premise is broken.** The original plan's Green pseudocode read `source.labels.filter(l => l.startsWith('kw:'))` to discover the source card's terms. There are NO `kw:*` labels on cards. The constant `keywordLabel(term) → 'keyword:<normalized>'` exists at `labels.ts:133` but is never called outside the label-unit test (grep: zero production callers). Keyword data lives in the sqlite `keyword_entries` table as a term → `entry_points` index, not as labels on cards. The original approach would always return `[]`. REVIEW option (b) was selected: change the signature so the caller passes terms in — the cascade Gate B callsite has them in scope from Pass 2/3 extraction.

### filterByKeywordOverlap (new export in keyword-index.ts)

```ts
// apps/silmari-mcp/src/lib/keyword-index.ts (additions — reuses module's existing imports)
import { brList, brShow } from './br-adapter.js';
import { fzLabelFromAddress } from './labels.js';
import type { BeadRow } from './navigate.js';

/**
 * Return every bead whose keyword-index entry overlaps with `terms` in at
 * least `minOverlap` terms. Sorted by overlap count DESC.
 *
 * The caller passes the source's terms in directly (usually produced by
 * cascade Pass 2/3 extraction). This avoids a reverse-lookup path that the
 * current keyword index does not support — there is no card → terms index.
 *
 * Graceful degradation (matches neighbor primitives):
 *   - `terms.length === 0`      → returns `[]`
 *   - `minOverlap > terms.length` → returns `[]` (no candidate can match)
 *   - entry resolves to no bead → skipped (not thrown)
 */
export function filterByKeywordOverlap(
  terms: string[],
  minOverlap: number = 2,
): BeadRow[] {
  if (terms.length === 0) return [];
  if (minOverlap > terms.length) return [];

  // Tally how many distinct input terms each entry appears in.
  // Key = raw entry string (bead id OR slash-form address — see
  // substrate-ambiguity note in the plan).
  const counter = new Map<string, number>();
  for (const term of terms) {
    const entry = lookupKeyword(term);
    if (!entry) continue;
    // Dedupe entries within a single term (shouldn't happen, but cheap).
    const seen = new Set<string>();
    for (const ep of entry.entry_points) {
      if (seen.has(ep)) continue;
      seen.add(ep);
      counter.set(ep, (counter.get(ep) ?? 0) + 1);
    }
  }

  // Threshold + sort (ties broken by insertion order via Array.from stability).
  const qualified: Array<[string, number]> = [];
  for (const [entry, count] of counter) {
    if (count >= minOverlap) qualified.push([entry, count]);
  }
  qualified.sort((a, b) => b[1] - a[1]);

  // Resolve entries → BeadRow. Tolerates BOTH bead-id and address shapes
  // until the substrate ambiguity is resolved upstream.
  const out: BeadRow[] = [];
  for (const [entry] of qualified) {
    const byId = brShow('idea', entry);
    if (byId) { out.push(byId as BeadRow); continue; }
    // Fall back: entry may be a slash-form address like "2/3a1".
    const rows = brList({ box: 'idea', labels: [fzLabelFromAddress(entry)], limit: 1 });
    if (rows.length > 0) out.push(rows[0] as BeadRow);
    // else: skip silently (entry points to a bead that no longer exists)
  }
  return out;
}
```

### Observable behaviors

1. `filterByKeywordOverlap([], 2)` returns `[]`.
2. `filterByKeywordOverlap(['voice'], 2)` returns `[]` (single-term input can never hit `minOverlap=2`).
3. `filterByKeywordOverlap(['voice', 'shame'], 2)` returns ONLY beads whose id appears in BOTH terms' `entry_points`.
4. Result order is sorted by overlap count DESC (higher overlap first).
5. When an `entry_points` value does not resolve to any bead, it is silently skipped — the function does not throw.
6. `hubMembers(hubId)` returns all cards attached to the hub via `ref:derives-from:<hubId>` edges (alias correctness — matches `listHubConstituents` output exactly).
7. `hubMembers(nonexistentHubId)` returns `[]` (inherited graceful-degradation behavior from `listInbound` / `brList`).

## Risk register

1. **Resolution-path performance at scale.** `brShow` + fallback `brList` per entry is O(entry_points) subprocess calls. For a hub-sized cascade (tens of entries per term, a few terms per call), this is fine. If a caller ever passes hundreds of terms, the subprocess fan-out dominates. Mitigation: measure before optimizing. If a hot path emerges, swap the fallback to a single `scanTrunk(trunk).beadsBySequence.get(sequence)` pass. Out of scope for Step 5.5.

2. **Dependency on 7h6 uncap.** `filterByKeywordOverlap` reads `lookupKeyword(term).entry_points`. While `MAX_ENTRY_POINTS = 4` (the cap 7h6 removes) is still in place, each term can index at most 4 cards — so an overlap query can return at most 4 × terms results, and most under-return for high-frequency terms. f23 is functionally correct without 7h6 but behaviorally weak until it lands. Implementer must check `bd deps f23` before declaring the slice ready; if 7h6 hasn't merged, document the under-return in the cascade Gate B callsite.

3. **Substrate ambiguity in `entry_points` (id vs address).** The resolution fallback (`brShow → brList` via `fzLabelFromAddress`) covers both shapes today. If a writer ever produces a malformed entry (neither an id nor a parseable address), the fallback silently skips it — which is correct for Step 5.5 but may mask a real data bug. Mitigation: the `fzLabelFromAddress` call throws on a malformed address (it does a `.replace('/', '_')` on the input — a malformed input produces a malformed label, not an exception). Verify this is tolerable in Behavior 5; if not, wrap the fallback in a try/catch that logs to stderr and skips.

4. **Empty hubs on deployed alpha.** Cosmic-migrated cards carry zero `ref:derives-from:*` edges on ionos01 today (see memory entry `feedback_silmari_graph_edge_recovery.md`). `hubMembers` will return `[]` for every migrated hub until the cosmic edge backfill runs. This is a deploy-side precondition, not a plan-internal bug — document it in the rollout notes and skip the alpha smoke assertion until backfill is complete.
